# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Layout note

The working directory is `TREASURY_DOGE/`, but the actual application and the git repo live in `alcohol-label-verification/` (the repo root) — run all commands from there. The repo now also tracks the supporting material that used to sit beside it in the parent folder: `reference/` (the take-home brief and the TTB labeling checklists — wine, malt beverage, distilled spirits — that are the source of truth for the compliance rules implemented in code), `tools/` (the PDF generator), `archive/`, and this `CLAUDE.md` itself. All four are kept out of the Vercel bundle via `.vercelignore`. Note the case split: `reference/` (uppercase-free name) holds the inputs, while `docs/` (lowercase) holds the submission outputs — on macOS's case-insensitive filesystem a `Docs/` would collide with `docs/`, which is why the brief was moved in under `reference/`. A one-line pointer `CLAUDE.md` remains at `TREASURY_DOGE/` (untracked, outside the repo) so the project instructions still auto-load when Claude Code is launched from the parent.

The repo is flat: `app/`, `tests/`, `evals/`, `scripts/` live at the repo root (there is no longer a `backend/` wrapper). Tests import `app.*`, so run them from the repo root. `docs/` (repo, lowercase) holds the submission documents (Design-Document.md, Design-Document.pdf, README.pdf) and is excluded from the Vercel bundle; they are regenerated by `tools/make_pdfs.py` (now inside the repo, run from the repo root: `venv/bin/python tools/make_pdfs.py`).

## Commands

All commands assume the venv at `alcohol-label-verification/venv/` and run from the repo root (`alcohol-label-verification/`). The venv python is invoked directly — it is not activated.

```bash
# Setup
cp .env.example .env      # then set GEMINI_API_KEY
python -m venv venv && venv/bin/python -m pip install -r requirements.txt
venv/bin/python -m pip install -e ".[dev]"   # dev extra: ruff + pillow (evals)

# Run the server
venv/bin/python -m uvicorn app.main:app --reload
# UI at http://localhost:8000/  ·  Swagger at http://localhost:8000/docs

# Tests (from the repo root — they import `app.*`)
venv/bin/python -m unittest discover tests

# Single test
venv/bin/python -m unittest tests.test_validation.ValidationTests.test_parse_abv_and_wine_path

# Lint (config in pyproject.toml — E/F/I/W/B/UP, line-length 120; same scope as CI)
venv/bin/ruff check app tests scripts evals
venv/bin/ruff check --fix app tests scripts evals

# Extraction-accuracy eval — each case is one live Gemini call, so needs GEMINI_API_KEY
# Default --schema unified mirrors the production Auto path (all-fields schema) and ALSO
# scores classify_category/classify_origin; it reports spurious-field extractions too.
venv/bin/python -m evals.run_eval               # all cases
venv/bin/python -m evals.run_eval --limit 5     # cheap partial run
venv/bin/python -m evals.run_eval --case napa-cabernet-vintage-sulfites   # one case (repeatable)
venv/bin/python -m evals.run_eval --prompt-file /tmp/p.txt      # A/B a candidate prompt without editing prompts.py
venv/bin/python -m evals.run_eval --schema focused   # legacy category-scoped schema (pre-unified baselines)

# Verify Gemini connectivity / API key
venv/bin/python -m scripts.gemini_smoke_test
```

The actual model is set in [clients.py](alcohol-label-verification/app/clients.py) (`GEMINI_MODEL`, currently `gemini-2.5-flash`; overridable via the `GEMINI_MODEL` env var). Extraction runs with thinking disabled (`thinking_budget=0`) and temperature 0 — that is what keeps it at ~2s against the ~5s product bar (thinking on measured ~7s); don't re-enable it casually. `GEMINI_TIMEOUT_MS` (default 25000, raised from 12s after live 504s on large files) must never be set below 10000 — the Gemini API rejects client deadlines under 10s with `400 INVALID_ARGUMENT`.

## Architecture

A FastAPI app that automates what TTB reviewers do by hand: comparing label artwork against an approved COLA application. The flow is a **two-stage pipeline**, and the two stages are deliberately decoupled.

**Stage 1 — Extraction** ([extraction.py](alcohol-label-verification/app/extraction.py)): Gemini Vision reads uploaded label image(s) and returns the regulated fields as JSON. Almost all domain knowledge about *how to read a label* lives in the prompt ([prompts.py](alcohol-label-verification/app/prompts.py)) — the hard distinctions between brand name vs. fanciful name vs. class/type, stripping "Imported"/"Domestic" from class/type, etc. When changing extraction behavior, edit the prompt, not the code — then re-run the full eval (prompt tweaks here have a history of backfiring on non-target cases). Extraction is schema-scoped: a known category restricts the response schema to its fields; `None` (the auto path) reads all 22. Gemini calls are retried up to 3×, the response is defensively parsed (fence-stripping, empty/JSON/shape checks, each failure mapped to a distinct `GEMINI_*` error code), and the whole retry loop is wall-clock-bounded: attempt 1 always runs, retries only start while `GEMINI_DEADLINE_SECONDS` budget remains (default 26s) and are cut off (`asyncio.wait_for`) when it expires — total worst case ≈ the deadline budget (~26s: one slow 25s per-call timeout plus a budget-capped retry), not attempts × timeout. One exception to "edit the prompt, not the code": extraction.py also strips a leading "Imported"/"Domestic" from `class_type` in code (`_ORIGIN_PREFIX`) as a backstop to the prompt rule — keep both layers in sync.

**Auto-detect** ([classify.py](alcohol-label-verification/app/classify.py)): deterministic, rule-based classification of product category and origin from already-extracted fields — no model call. Category matches word-boundary keywords against `class_type` first (falling back to `fanciful_name`), breaking word-level ties toward the right-most keyword because TTB designations end with the class noun ("Bourbon Barrel **Stout**" is a stout). Never use plain substring matching here — that's how "Porter" became wine ("port") and "Ginger Beer" became spirits ("gin"). With no keyword match at all the fallback is `malt_beverage` — the auto path never errors on an unrecognizable label, it guesses. Origin is "imported" iff an importer or country of origin is present. `/extract` and `/verify` default `product_category`/`origin_type` to `"auto"`, which triggers all-fields extraction + classification; any value that is neither `"auto"` nor a valid enum value is a 422, not a silent fallback.

**Stage 2 — Validation** ([validation.py](alcohol-label-verification/app/validation.py)): pure, rule-based field comparison with no I/O — this is the core domain logic and the most-tested module. The central abstraction is the **field-requirement matrix**: every field's status (required / conditional / optional) is computed as `COMMON_REQUIRED_FIELDS + PRODUCT_REQUIRED_FIELDS[category] + ORIGIN_REQUIRED_FIELDS[origin]`. All three product categories route through one engine, `_validate_by_requirements`; the per-category `validate_*` functions are thin wrappers. Special cases:
- `government_warning` — checked against the exact statutory text on BOTH sides: its validation entry is a **dict** `{"expected": <code>, "label": <code>}` (every other field is a status string — any new consumer must handle this). Codes are granular (`FAIL_MISSING_HEADING`, `FAIL_HEADING_FORMAT`, `FAIL_TEXT_MISMATCH`, `MISSING`), whitespace/OCR line-breaks normalized away, punctuation and the ALL-CAPS heading enforced.
- wine `alcohol_content` — drives `get_wine_path` (the <7% vs ≥7% ABV branching TTB rules hinge on).
- `brand_name` — the deemed-brand fallback (27 CFR 4.33, brand = bottler name) applies only when the *expected* brand is also empty; a COLA brand with a brandless label is `MISSING`.

The status vocabulary goes beyond PASS/FAIL/MISSING (`DEEMED_FROM_BOTTLER`, `EXEMPT_TABLE_WINE`, `PRESENT_ON_LABEL_CONFIRM_APPLICABILITY`, `FAIL_APPELLATION_REQUIRED_BY_TRIGGER`, `FAIL_NOT_ALLCAPS`, `EXPECTED VALUE MISSING`, `NOT REQUIRED`), and app.js hardcodes a display label + verdict class per code — a new status code in validation.py needs a matching app.js entry or the UI renders it raw. validation.py also exposes `compute_label_checks` — label-only advisory checks (net-contents unit system, wine/spirits standard of fill, origin coherence, state-of-distillation vs address) returned as `compliance_checks` in verify responses. Two of its behaviors are domain decisions, not bugs: standard-of-fill is deliberately advisory (the CFR size tables change by regulation), and a conditional field present on the label but blank in the expected COLA is flagged for the reviewer rather than passed (additive/sulfite applicability is a COLA formulation fact, not visible in the artwork).

Normalization is intentionally lenient so equivalent values PASS ("STONE'S THROW" vs "Stone's Throw" must not flag): lowercasing, punctuation stripping, state/country folding (multi-word names fold before single-word — "West Virginia" must not become "West VA"), compound address lead phrases ("Distilled and Bottled by") stripped, and volume parsing that sums chained customary measures ("1 PINT 0.9 FL OZ" ≈ 499.8 mL) and prefers the metric half of dual statements. Edit `normalize_field`/`parse_volume` when a real-world equivalent value is flagged as a mismatch.

**Correct-and-compare model** ([main.py](alcohol-label-verification/app/main.py)): the UI is a single editable "Label application fields" form. Extraction pre-fills it (and the hidden reviewed side) from one Gemini call — a documented simplification; there is no separate COLA upload. The reviewer corrects the form to match the approved application, then `/verify-reviewed` re-runs validation **without another Gemini call**. This is why validation is pure and separate from extraction.

**Endpoints**: `/extract` (fields + resolved category/origin), `/verify` (extract + validate in ONE Gemini call — used by batch mode so each row costs one request), `/verify-reviewed` (validate corrected fields, no Gemini) — these three are the cost-bearing endpoints and share `COST_GUARDS` (rate limit + optional bearer auth). All five documented endpoints have Pydantic `response_model`s ([schemas.py](alcohol-label-verification/app/schemas.py)); `/verify-reviewed`'s `LabelFields` forbids unknown keys (typo'd field names 422 instead of validating against empty). Plus `/fields` and `/field-requirements` (drive the dynamic UI field lists), `/health` + `/readyz` (readiness checks the API key without calling Gemini), and `/`, `/how-to` (static pages).

**Field set is defined once** ([fields.py](alcohol-label-verification/app/fields.py)): `FIELD_SPECS` (22 fields) is the single source of truth — keys, display labels, control types, order (tracking COLA Form 5100.31 box order), and which categories each applies to. The Pydantic model (`schemas.LabelFields`), the validation field order (`validation.FIELD_ORDER`), the extraction response schema, the UI (via `GET /fields`), **and the hardcoded fallback `FIELD_CONFIG` at the top of app.js** all derive from it. Add or rename a regulated field in `FIELD_SPECS`, then mirror it in the app.js fallback list.

**Upload security** ([uploads.py](alcohol-label-verification/app/uploads.py)): every upload is checked three ways before reaching Gemini — file extension, browser content-type, and magic-byte signature must all agree. 10 MB cap per file, streamed in chunks. PDF/PNG/JPEG/WebP only. Two documented *accepted* residual risks (don't re-flag or "harden" without cause): the magic-byte check is prefix-only (polyglots pass) and PDFs aren't parsed for page count/decompression bombs — acceptable because uploaded bytes are only forwarded to Gemini, never decoded or executed locally.

**API hardening** ([security.py](alcohol-label-verification/app/security.py)): open by default, lockable for production entirely via env vars. Per-IP sliding-window rate limit (default 120/min, sized for the brief's 200–300-label batch; 429s carry a `Retry-After` header the frontend's batch pacing keys off). Rate-limit state is an in-memory per-process dict — explicitly single-instance; on Vercel each serverless instance keeps its own bucket, so the effective live limit differs from local. The client IP is the **right-most** valid `X-Forwarded-For` entry (left entries are client-spoofable), and the tracking dict is bounded. The body cap is a pure ASGI middleware that counts streamed bytes, so chunked requests can't bypass it (`MAX_REQUEST_BYTES`, default 25 MB total — a second ceiling on top of the per-file 10 MB upload cap); its 413 uses the standard envelope with `request_id`. CSP includes `frame-src 'self' blob:` (the in-app PDF preview iframe breaks without it) and allows **no inline scripts** — the theme bootstrap is an external file (`theme-init.js`) on purpose; an inline `<script>` or event handler in the HTML will be silently blocked. Optional `APP_API_TOKEN` bearer auth (constant-time compare), CORS allowlist, Swagger kill-switch (`ENABLE_DOCS`). See `.env.example` for the full surface. Request correlation IDs (`X-Request-ID`, validated against `^[A-Za-z0-9._-]{1,64}$`, propagated into logs and error responses) live in [observability.py](alcohol-label-verification/app/observability.py). Middleware order in main.py is load-bearing: `RequestIdMiddleware` is added **last** so it runs outermost — that is the only reason the body-cap 413 carries a `request_id`; reordering the `add_middleware` calls silently drops it.

**Errors** ([errors.py](alcohol-label-verification/app/errors.py)): all failures raise `AppError` (which can carry response headers, e.g. `Retry-After`), rendered by one handler into `{"error": {code, message, details, request_id}}`. Add new failure modes as `AppError`s with a distinct `code`, not bare HTTP exceptions.

**Frontend**: a single static no-build bundle ([static/](alcohol-label-verification/app/static/)) served by FastAPI itself. Single-label mode auto-detects category/origin: the Category/Origin selects **stay on value `"auto"`** — detection is shown by rewriting the Auto option's text ("Auto — detected: Wine") and is cleared on any file change; never write a detected value into the select (that silently turns detection into a manual constraint for the next upload). Batch ("Multiple labels") mode gives each file a per-row Pass/Needs-attention verdict via `/verify`, pairs front/back images by `_front`/`_back` filename, paces itself on 429 `Retry-After` with a shared pause gate, and patches rows in place (`updateBatchRow`) rather than rebuilding the table. Theming is two independent axes — theme (light/dark toggle) × layout (standard/government select), persisted under separate localStorage keys and applied pre-paint by `theme-init.js` (which also migrates the legacy single-value "government" theme to layout=government) — giving four palette blocks in styles.css, each with its own on-accent ink tokens (`--on-primary`, `--on-green`, …) — never hardcode `#fff` on an accent background; dark-theme accents are light pastels. API tests assert specific strings in `index.html` / `app.js` / `how-to.html`, so UI text changes can break tests — run the suite after copy edits.

**Evals** ([evals/](alcohol-label-verification/evals/)): the accuracy gate for the whole tool — `run_eval.py` runs each ground-truth case in `cases.json` through the *production* extraction path and scores fields with the same `validation.match_field` used for live verdicts. The default `--schema unified` measures the shipped Auto path: all-fields extraction plus `classify_category`/`classify_origin` parity against the case's known category/origin. Spurious-field detection flags non-empty extractions absent from ground truth (fails the case; the headline field-accuracy denominator is unchanged for comparability). Artwork is synthetic (rendered text via `render.py`), so a green eval validates the prompt's field-separation logic, **not** OCR robustness on real photos. Change extraction behavior → re-run the full eval before trusting it. Baseline: 100% (418/418), 32/32 cases, p95 well under the 5s product bar. Each run overwrites `evals/last_results.json` — the machine-readable baseline artifact (per-field tallies, latency p50/p95, spurious, classification misses) to compare runs against; historical run logs and candidate prompts are archived outside the repo in `TREASURY_DOGE/archive/evals/`. `government_warning` is scored as verdict *parity* (`check_government_warning(got) == check_government_warning(exp)`), so deliberately non-compliant cases must be transcribed verbatim — a model that "helpfully" corrects the text fails the case.

**Tests** (`tests/`): `test_validation.py` (domain rules), `test_classify.py` (category/origin classification incl. the substring-collision suite), `test_api.py` (endpoint contracts, response shapes, UI string assertions, Gemini-mocked extraction paths), `test_platform.py` (rate limiting, body cap, XFF, request-ID, retry-budget contracts). Two `test_api.py` tests pin the extraction retry contract: calls==1 when the budget is 0, calls==3 within budget — keep that contract if touching the retry loop.

## Entrypoints & deployment

- Canonical app: `app.main:app`. Live URL: https://alcohol-label-verification-fawn.vercel.app
- Deployed on Vercel via [vercel.json](alcohol-label-verification/vercel.json) (routes everything to `app/main.py`) and [pyproject.toml](alcohol-label-verification/pyproject.toml). `GEMINI_API_KEY` must be set as a Vercel env var. `.vercelignore` keeps the venv, tests, scripts, and evals out of the bundle, and vercel.json's `config.includeFiles` ships `app/static/` into the function bundle — remove it and the deployed UI breaks while local still works. The user deploys manually (`npx vercel --prod` from the repo root) and pushes to GitHub separately — a green local tree does not mean the live URL or the repo is current.
- `requirements.txt` is what Vercel and CI actually install; `pyproject.toml` mirrors it for local editable installs — runtime dependency changes must land in **both**. CI (`.github/workflows/ci.yml`: ruff + unittest on pushes to main and PRs) runs with no venv, no dev extra (no Pillow), and no `GEMINI_API_KEY` — tests must pass without either. ruff is pinned to the same version in the dev extra and CI; keep them in sync.
- Diagnose live "Gemini rejected the request" reports with `scripts.gemini_smoke_test` (prints the real 4xx reason) or the Vercel function logs.
